1. OVERVIEW
--------------------------------------------------------------------------------
The project uses Firebase for:
  • Authentication (Firebase Auth): sign-in, roles, session
  • Database (Cloud Firestore): orders, menu, users, inventory, settings, logs
  • Backend logic (Cloud Functions): order creation, payment confirm/reject,
    QR validation, serve item, inventory updates (client cannot write orders/inventory)

The React frontend talks to Firebase Auth and Firestore for reads and to
Cloud Functions (Callables) for all critical writes when VITE_USE_CALLABLES
is not set to "false".

Backend (Node/Express in backend/) uses Firebase only for identity: it stores
firebase_uid in PostgreSQL and issues JWT; it does not use Firestore or
Firebase Admin in the current code paths.

4. AUTHENTICATION — HOW IT WORKS
--------------------------------------------------------------------------------
  File: services/auth.ts (Firebase Auth + Firestore users collection)

  4.1 Methods
  • signInWithGoogle() — GoogleAuthProvider; popup (desktop) or redirect (mobile)
  • signIn(email, password) — signInWithEmailAndPassword → getUserProfile → validate role/active
  • signUp(email, password, name, role?, studentType?) — createUserWithEmailAndPassword + createUserProfile
  • signOut() — Firebase signOut
  • signInAsGuest(name, studentType?) — returns in-memory profile (uid like guest_*); no Firebase Auth
  • onAuthStateChange(callback) — wraps onAuthStateChanged; handles getRedirectResult; callback(user, profile)
  • getUserProfile(uid) — getDoc(users/{uid}); role validated/inferred from email; optional Firestore fix
  • createUserProfile(uid, email, name, role, studentType?) — setDoc(users/{uid}, { merge: true })
  • updateUserRole(uid, role), toggleUserStatus(uid, active) — setDoc merge

  4.2 Profile storage
  • Collection: users
  • Document: users/{uid}
  • Fields: uid, name, email, role, studentType, active, createdAt, lastActive
  • Role inferred from email (admin@cse.com, cashier@cse.com, server@cse.com) if missing/wrong

  4.3 useAuth hook (hooks/useAuth.ts)
  • Subscribes to onAuthStateChange
  • State: user (FirebaseUser | null), profile (UserProfile | null), loading, role
  • role = profile?.role ?? null — used by App.tsx for routing

  Used in: App.tsx (routing), LoginView (signIn), WelcomeView (signInWithGoogle), logout flow


5. FIRESTORE — COLLECTIONS AND USAGE
--------------------------------------------------------------------------------
  All client access goes through services/firestore-db.ts (and services/reporting.ts for reports).
  Cloud Functions (functions/index.js) write orders, inventory, serveLogs, scanLogs, flags.

  5.1 users/{uid}
  • Read: auth.ts (getUserProfile), firestore-db (getUserProfile, listenToAllUsers)
  • Write: auth.ts (createUserProfile, updateUserRole, toggleUserStatus); rules allow own + admin
  • Used for: login, role, profile, admin user list

  5.2 menu/{itemId}
  • Read: listenToMenu, getDocs(menu) in initializeMenu
  • Write: addMenuItem, updateMenuItem, deleteMenuItem, initializeMenu (batch); rules: admin only
  • Used for: HomeView menu, Admin menu CRUD, initialize on first load

  5.3 orders/{orderId}
  • Read: getOrder, listenToOrder, listenToAllOrders, listenToUserOrders, listenToAllUserOrders,
    listenToLatestActiveQR, listenToPendingCashOrders, listenToActiveOrders, listenToPendingItems;
    reporting.ts (fetchReport)
  • Write: ONLY Cloud Functions (createOrder, confirmPayment, rejectPayment, validateQRCode, serveItem)
  • Client create/update disabled in rules (allow create: false, allow update: false)
  • Document shape: orderId, userId, userName, items[], totalAmount, paymentType, paymentStatus,
    orderStatus, qrStatus, qr?, cafeteriaId, createdAt, scannedAt, servedAt, confirmedBy, confirmedAt,
    rejectedBy, rejectedAt
  • Used for: PaymentView (create via callable, listen), QRView (listen), CashierView (listen + confirm/reject),
    ServingCounterView (validate QR + serve via callables), OrdersView, reporting

  5.4 inventory/{itemId}
  • Read: getInventory, listenToInventory; rules: admin/server/cashier
  • Write: ONLY Cloud Functions (serveItem increments consumed; updateInventory callable); rules: allow write: false
  • Used for: Admin inventory, serveItem in Functions

  5.5 inventory_meta/{itemId}, inventory_shards/{itemId}/shards/{shardId}
  • Defined in rules (read: staff; write: false). Reserved for future sharded counters.

  5.6 settings/global
  • Read: getSettings, listenToSettings; rules: read true
  • Write: updateSettings; rules: admin only
  • Used for: Admin dashboard, maintenance/announcement

  5.7 scanLogs/{logId}
  • Read: getScanLogs; rules: admin/server/cashier
  • Write: ONLY Cloud Functions (validateQRCode creates log); rules: allow create: false
  • Used for: Audit / server scan history

  5.8 serveLogs/{logId}
  • Read: getServeLogs; rules: admin/server/cashier
  • Write: ONLY Cloud Functions (serveItem creates log); rules: allow create: false
  • Used for: Audit / serving history

  5.9 dailyReports/{reportId}
  • Read: getDailyReport; rules: admin/cashier
  • Write: admin only
  • Used for: Reporting (optional)

  5.10 carts/{userId}
  • Read/Write: saveCartDraft; rules: own userId only
  • Used for: Optional cart persistence (HomeView)

  5.11 flags/{flagId}
  • Read: admin only; Write: only Cloud Functions (onOrderCreated fraud trigger)
  • Used for: Fraud detection (high order rate)


6. CLOUD FUNCTIONS (functions/index.js)
--------------------------------------------------------------------------------
  Region: us-central1 (client uses getFunctions(app, "us-central1") in services/callables.ts)

  • createOrder — Callable. Creates order in Firestore. Allows authenticated user or guest (data.userId starting with "guest_").
  • confirmPayment — Callable. Cashier/admin. Sets paymentStatus SUCCESS, qrStatus ACTIVE, generates QR, confirmedBy/confirmedAt.
  • rejectPayment — Callable. Cashier/admin. Sets paymentStatus REJECTED, orderStatus CANCELLED, qrStatus REJECTED.
  • validateQRCode — Callable. Parses QR payload, verifies HMAC (QR_SECRET_KEY), updates order to qrStatus USED, orderStatus ACTIVE, writes scanLog.
  • serveItem — Callable. Server/admin. Updates order items (servedQty, remainingQty), orderStatus COMPLETED if all served, writes serveLog, increments inventory.consumed.
  • updateInventory — Callable. Admin. Create/update inventory doc (itemName, openingStock, consumed, category).
  • cancelOrder — Callable. User (own order) or admin/cashier. Sets order CANCELLED / REJECTED.
  • onOrderCreated — Firestore trigger on orders/{orderId} onCreate. Flags user if >10 orders in last hour (fraud).

  QR verification in Functions uses same HMAC-SHA256 + base64url as client (services/qr.ts). Secret: process.env.QR_SECRET_KEY or default string.


7. CLIENT — CALLABLES VS FIRESTORE
--------------------------------------------------------------------------------
  File: services/callables.ts
  • getFunctions(app, "us-central1")
  • connectFunctionsEmulator(functions, "localhost", 5001) when VITE_USE_FUNCTIONS_EMULATOR=true in dev
  • Exports: createOrderCallable, confirmPaymentCallable, rejectPaymentCallable, validateQRCodeCallable,
    serveItemCallable, updateInventoryCallable, cancelOrderCallable, useCallables()

  File: services/firestore-db.ts
  • useCallables() = (import.meta.env.VITE_USE_CALLABLES !== "false")
  • When useCallables() is true:
    - createOrder → createOrderCallable (payload: userId, userName, items, totalAmount, paymentType, paymentStatus, cafeteriaId)
    - confirmCashPayment → confirmPaymentCallable({ orderId })
    - rejectCashPayment → rejectPaymentCallable({ orderId })
    - validateQRForServing → validateQRCodeCallable({ qrPayload }) then getOrder(orderId) and return Order
    - serveItem → serveItemCallable({ orderId, itemId, servedBy })
    - updateInventoryItem → updateInventoryCallable({ itemId, itemName?, openingStock?, consumed?, category? })
  • When useCallables() is false: same functions use direct Firestore (setDoc/updateDoc/runTransaction). With current rules, orders/inventory client writes are disallowed, so production must use callables or relax rules.


8. REAL-TIME LISTENERS (FIRESTORE)
--------------------------------------------------------------------------------
  All in firestore-db.ts; return unsubscribe function.

  • listenToMenu — collection(menu), filter active, sort name in memory
  • listenToOrder(orderId) — doc(orders, orderId)
  • listenToAllOrders — orders orderBy createdAt desc
  • listenToUserOrders(userId) — userId == X, paymentStatus in [SUCCESS, PENDING]
  • listenToAllUserOrders(userId) — userId == X
  • listenToLatestActiveQR(userId) — userId, paymentStatus SUCCESS, qrStatus ACTIVE; then latest by createdAt
  • listenToPendingCashOrders — paymentType CASH, paymentStatus PENDING, orderBy createdAt desc
  • listenToActiveOrders — orderStatus ACTIVE, paymentStatus SUCCESS, qrStatus USED, orderBy scannedAt desc
  • listenToPendingItems — same query as listenToActiveOrders; flattens items with remainingQty > 0
  • listenToAllUsers — collection(users); sort by createdAt in memory
  • listenToInventory — collection(inventory)
  • listenToSettings — doc(settings, global)


9. REPORTING (services/reporting.ts)
--------------------------------------------------------------------------------
  • Imports: collection, getDocs, query, where, orderBy from firebase/firestore; db from firebase
  • fetchReport({ role, start, end }) — queries orders by createdAt range and paymentStatus; 5-min cache
  • computeReport — aggregates revenue, payment split, item sales, trend, peak hours
  • invalidateReportsCache — called after createOrder, confirmCashPayment, rejectCashPayment
  • exportReport — PDF/CSV/XLSX/JSON/PNG from report data


10. SECURITY RULES (firestore.rules) — SUMMARY
--------------------------------------------------------------------------------
  • users: read own or admin; create/update own or admin; delete admin
  • menu: read all; write admin
  • orders: read all; create false; update false; delete admin (only Functions write)
  • inventory: read admin/server/cashier; write false (only Functions)
  • inventory_meta, inventory_shards: read staff; write false
  • settings: read all; write admin
  • scanLogs: read staff; create false; update/delete admin
  • serveLogs: read staff; create false; update/delete admin
  • dailyReports: read admin or cashier; write admin
  • carts: read/write own userId
  • flags: read admin; write false (Functions write)


11. FIRESTORE INDEXES (firestore.indexes.json)
--------------------------------------------------------------------------------
  Orders composite indexes:
  • paymentType ASC, paymentStatus ASC, createdAt DESC
  • orderStatus ASC, paymentStatus ASC, qrStatus ASC, scannedAt DESC
  • userId ASC, paymentStatus ASC, createdAt DESC
  • createdAt ASC, paymentStatus ASC
  • userId ASC, createdAt ASC
  • orderStatus ASC, createdAt DESC

  (serveLogs and scanLogs use single-field indexes; no composite in file.)
